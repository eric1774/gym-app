**Tests:**

1. `getWorkoutDaysForMonth(2026, 3)` — The function builds a wide UTC date range (one day buffer on each side of the month), queries sessions, then filters by local date to the exact month.
   Mock executeSql returning 3 sessions:
   - {id: 1, completed_at: '2026-03-05T18:00:00Z'}
   - {id: 2, completed_at: '2026-03-05T20:00:00Z'} (same day as #1)
   - {id: 3, completed_at: '2026-03-10T15:00:00Z'}
   Mock getLocalDateString to return '2026-03-05' for the first two dates and '2026-03-10' for the third.
   Use `mockGetLocalDateString.mockImplementation((d: Date) => { ... })` to map input dates to output strings deterministically.
   Assert returned CalendarDaySession array: [{date:'2026-03-05', sessionCount:2}, {date:'2026-03-10', sessionCount:1}] sorted ascending.

   Edge case: session that falls in the buffer zone but whose local date is in the previous month should be excluded. E.g., a session with completed_at on March 1 00:30 UTC whose local date (e.g. UTC-5) is Feb 28 — should not appear in March results. Mock getLocalDateString to return '2026-02-28' for that date, assert it's filtered out.

   Edge case: no completed sessions — returns `[]`.

2. `getFirstSessionDate()` — mock returns {earliest: '2026-01-15T10:00:00Z'}. Mock getLocalDateString(new Date('2026-01-15T10:00:00Z')) returns '2026-01-15'. Assert returns '2026-01-15'. Edge case: no completed sessions (earliest is null) returns null. Edge case: empty ResultSet returns null.

3. `getDaySessionDetails('2026-03-15')` — **Complex, multi-step:**
   - First SELECT returns all completed sessions. Mock returns 2 session rows:
     {id:1, started_at:'2026-03-15T09:00:00Z', completed_at:'2026-03-15T10:00:00Z', program_day_id:1, program_day_name:'Push'}
     {id:2, started_at:'2026-03-14T09:00:00Z', completed_at:'2026-03-14T10:00:00Z', program_day_id:2, program_day_name:'Pull'}
   - Mock getLocalDateString: for session 1's completed_at returns '2026-03-15' (matches), for session 2's returns '2026-03-14' (excluded).
   - For matching session 1: SELECT sets returns 2 rows:
     {set_number:1, weight_kg:100, reps:8, is_warmup:0, exercise_id:1, exercise_name:'Bench Press'}
     {set_number:2, weight_kg:110, reps:6, is_warmup:0, exercise_id:1, exercise_name:'Bench Press'}
   - For PR detection: SELECT max volume for exercise_id=1 in prior sessions returns {max_volume: 700} (100*7=700). Set 1 volume=800, so isPR=true; set 2 volume=660, isPR=false.
   - Assert returned CalendarSessionDetail: sessionId=1, durationSeconds=3600, totalSets=2, totalVolume=100*8+110*6=1460, exerciseCount=1, prCount=1, exercises[0].sets[0].isPR=true, exercises[0].sets[1].isPR=false.

   Edge case: warmup sets excluded from totalSets and totalVolume. A warmup set (is_warmup=1) should have isPR=false regardless. Mock a row with is_warmup=1, assert it's counted in exercises but not in totalSets/totalVolume/prCount.

   Edge case: no prior sessions for PR check (max_volume is null) — isPR=false for all sets (consistent with first-ever-not-a-PR logic).
