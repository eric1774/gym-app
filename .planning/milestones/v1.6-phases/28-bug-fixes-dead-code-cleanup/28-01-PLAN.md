# Phase 28, Plan 1: Bug Fixes & Dead Code Cleanup

## Objective

Fix all functional bugs and remove dead code identified in the v1.6 milestone audit:
unpair properly disconnects BLE, below-zone BPM renders correctly, and dead
types/exports are removed.

## Tasks

### Task 1: Fix handleUnpair in SettingsScreen — call disconnect() first

**Type:** auto
**File:** `src/screens/SettingsScreen.tsx`

The `handleUnpair` callback calls `clearPairedDevice()` from HRSettingsService
directly, bypassing the BLE disconnect. This leaves the BLE GATT connection open.

Fix: Replace `clearPairedDevice()` call with `disconnect()` from `useHeartRate()`,
which already calls `clearPairedDevice()` internally and also cancels the GATT
connection.

Update the import — remove `clearPairedDevice` from HRSettingsService import,
and add `disconnect` to the `useHeartRate()` destructure.

**Done when:**
- `handleUnpair` calls `disconnect()` (not `clearPairedDevice()` directly)
- `clearPairedDevice` is no longer imported from HRSettingsService in SettingsScreen
- `disconnect` is destructured from `useHeartRate()`
- Tests updated to mock `disconnect` instead of `clearPairedDevice`

### Task 2: Fix getHRZone — return null below Zone 1 threshold

**Type:** auto
**File:** `src/utils/hrZones.ts`

`getHRZone` currently clamps BPM below 50% of maxHr to Zone 1. The correct
behavior is to return `null` when BPM is below Zone 1 (below-zone BPM is not
in any zone — render without zone color or label).

Fix: Change return type to `HRZoneInfo | null`. Replace the Zone 1 clamp with
`return null`.

Update `src/utils/__tests__/hrZones.test.ts` to assert that below-zone BPM
returns `null`.

**Done when:**
- `getHRZone` returns `null` for BPM < 50% of maxHr
- Return type is `HRZoneInfo | null`
- Test updated to assert `null` return for below-zone BPM

### Task 3: Update WorkoutScreen bpmZone for null-returning getHRZone

**Type:** auto
**File:** `src/screens/WorkoutScreen.tsx`

The `bpmZone` useMemo calls `getHRZone` which now returns `null` for below-zone
BPM. The existing JSX already handles `bpmZone = null` by using
`bpmZone ? bpmZone.color : colors.primary` — so the display logic is correct.

The dead code to remove: the `effectiveAge` fallback `age ?? 35` is unnecessary
because the guard `if (age === null && override === null) return null;` already
ensures we never reach `computeMaxHR` with `age === null` when `override` is
also null. When `override !== null`, `age` is not used at all.

Fix: Replace `const effectiveAge = age ?? 35; const maxHr = computeMaxHR(effectiveAge, override);`
with `const maxHr = computeMaxHR(age!, override);` — safe because the guard
above ensures `age` is not null when we reach this line (if override is null, then
age must not be null; if override is not null, age is not used).

Wait — actually re-check: the guard is `if (age === null && override === null) return null`.
This means: if BOTH are null, return null. But if `age === null` and `override !== null`,
we proceed. In that case `computeMaxHR(null!, override)` would be called — but
`computeMaxHR` returns `override` when override is not null, so `age` is never used.
The `age ?? 35` fallback handles this by providing 35 as a dummy value — but it's
confusing dead code because the 35 is never actually used in the formula.

Fix: Change the guard to `if (age === null && override === null) { return null; }`
(already done), then use `const maxHr = computeMaxHR(age ?? 0, override);` — the
`?? 0` is not used when override is set, making intent clearer than `?? 35`.

**Done when:**
- The `effectiveAge` variable is removed from WorkoutScreen
- `computeMaxHR` is called with `age ?? 0` (or directly with a non-fallback)
- No behavioral change to zone rendering (null zone stays null)

### Task 5: Remove dead getComputedMaxHR export from HRSettingsService

**Type:** auto
**Files:** `src/services/HRSettingsService.ts`, `src/screens/SettingsScreen.tsx`,
`src/services/__tests__/HRSettingsService.test.ts`

`getComputedMaxHR` is exported from `HRSettingsService.ts` but:
1. It is imported in `SettingsScreen.tsx` but never called (SettingsScreen
   computes max HR inline with `Math.round(208 - 0.7 * age)`)
2. The same logic is already available via the synchronous `computeMaxHR` in
   `src/utils/hrZones.ts`

Fix:
1. Delete `getComputedMaxHR` function from `HRSettingsService.ts`
2. Remove `getComputedMaxHR` from the import in `SettingsScreen.tsx`
3. Remove the `getComputedMaxHR` test suite from `HRSettingsService.test.ts`
   (the mock can stay as-is or be removed)
4. Remove `getComputedMaxHR` from the mock in `SettingsScreen.test.tsx`

**Done when:**
- `getComputedMaxHR` is not defined in HRSettingsService.ts
- `getComputedMaxHR` is not imported in SettingsScreen.tsx
- Test file no longer has `getComputedMaxHR` test cases
- SettingsScreen test mock no longer includes `getComputedMaxHR`
