# Phase 26 Plan 01: HR Data Persistence

## Objective

Buffer HR samples in memory during a workout and batch-flush them to SQLite on session end. Compute avg/peak HR from the buffer, persist to workout_sessions, and surface them on the workout summary card and calendar day detail view.

## Context

- @.planning/phases/26-hr-data-persistence/26-CONTEXT.md
- @src/context/HeartRateContext.tsx
- @src/db/sessions.ts
- @src/db/calendar.ts
- @src/types/index.ts
- @src/screens/WorkoutScreen.tsx
- @src/screens/CalendarDayDetailScreen.tsx

## Tasks

### Task 1: Update WorkoutSession type and DB mapper

**Type:** auto

Add `avgHr: number | null` and `peakHr: number | null` to the `WorkoutSession` TypeScript interface. Update `rowToSession` in sessions.ts to map `avg_hr` and `peak_hr` columns.

**Done criteria:**
- WorkoutSession interface has avgHr and peakHr fields (nullable)
- rowToSession maps avg_hr -> avgHr and peak_hr -> peakHr
- All existing callers still compile (fields are optional/nullable)

### Task 2: Add HR DB functions to sessions.ts

**Type:** auto

Add two new functions to `src/db/sessions.ts`:
1. `flushHRSamples(sessionId, samples)` — batch-INSERT all samples into heart_rate_samples in a single transaction. Each sample is `{bpm: number, recordedAt: string}`.
2. `updateSessionHR(sessionId, avgHr, peakHr)` — UPDATE workout_sessions SET avg_hr, peak_hr WHERE id = sessionId.

**Done criteria:**
- flushHRSamples inserts all samples in one executeSql call using a multi-value INSERT or loop within a transaction
- updateSessionHR persists avg and peak HR to the session row
- Functions are exported from sessions.ts

### Task 3: Add sample buffering to HeartRateContext

**Type:** auto

Extend `HeartRateContext.tsx`:
1. Add a `hrSamplesRef = useRef<{bpm: number; recordedAt: string}[]>([])` to buffer samples
2. In the HR characteristic monitor callback, whenever a valid BPM is parsed, push `{bpm, recordedAt: new Date().toISOString()}` to hrSamplesRef.current
3. Add a `flushHRSamples(sessionId: number): Promise<void>` function to the context that: computes avg and peak from the buffer, calls db flushHRSamples + updateSessionHR, then clears the buffer
4. Expose `flushHRSamples` in the context value and HeartRateContextValue interface

The monitoring callback update must apply in all three places where `monitorCharacteristicForService` is called (connectToDevice, attemptAutoReconnect, and the reconnect scheduleReconnect).

**Done criteria:**
- hrSamplesRef accumulates {bpm, recordedAt} on every valid BPM notification
- flushHRSamples computes avg (Math.round of mean) and peak (Math.max), calls DB functions, clears buffer
- flushHRSamples is a no-op (does nothing) if buffer is empty
- flushHRSamples is exported from context value
- All three monitor callbacks push to the buffer

### Task 4: Wire flush into workout session end flow

**Type:** auto

In `WorkoutScreen.tsx`, update the "End Workout" handlers to call `flushHRSamples(session.id)` before `endSession()`. This applies in both alert paths (program workout and free workout).

The `flushHRSamples` function should be destructured from `useHeartRate()`.

**Done criteria:**
- Both onPress handlers call `await flushHRSamples(session.id)` before `await endSession()`
- The flush completes before the summary screen renders
- If no HR samples were buffered (no device connected), flush is a no-op and workflow is unchanged

Also update `CalendarSessionDetail` type in types/index.ts to add optional `avgHr: number | null` and `peakHr: number | null` fields.

Update `getDaySessionDetails` in calendar.ts to SELECT `avg_hr` and `peak_hr` from workout_sessions and include them in the returned CalendarSessionDetail objects.

**Done criteria:**
- getActiveSession SELECT includes avg_hr, peak_hr columns

### Task 6: Display HR stats on WorkoutSummary

**Type:** auto

Update `WorkoutSummary` component in WorkoutScreen.tsx:
1. Add `avgHr?: number | null` and `peakHr?: number | null` props to `WorkoutSummaryProps`
2. When summaryData is built (in the end-workout handler), query the session to get avg/peak HR after the flush completes, or pass them from the context
3. Render two stat rows conditionally — only when avgHr is not null:
   - "Avg HR" — value in BPM format (e.g. "142 bpm")
   - "Peak HR" — value in BPM format (e.g. "178 bpm")

**Implementation approach:** After `await endSession()`, query the completed session to get avg/peak HR by reading from the HeartRateContext's buffer before flushing. Actually the simpler approach: capture currentBpm context values aren't useful; instead, after flushHRSamples computes avg/peak, return them. Modify `flushHRSamples` to return `{avgHr: number | null, peakHr: number | null}` so the caller can store them in summaryData.

**Done criteria:**
- WorkoutSummaryProps has avgHr and peakHr optional props
- Avg HR and Peak HR stat rows render when avgHr is not null
- They are hidden (no empty rows) when no HR was recorded
- summaryData state includes avgHr and peakHr

## Success Criteria

1. HR samples accumulate in memory during a workout and are batch-inserted into heart_rate_samples in a single SQLite transaction when the session ends

4. The calendar day detail view displays average HR and peak HR for past sessions that recorded HR data
